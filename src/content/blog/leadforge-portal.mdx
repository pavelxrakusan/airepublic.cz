Data jsou v pipeline ([architektura tady](/blog/leadforge-architektura)). Ale surový dataset v databázi nikomu nepomůže. Potřeboval jsem portál — místo, kde si prodejce najde přesně ty SVJ, které chce oslovit.

Tak vznikl **leadforge.cz**.

## Tech stack

| Vrstva | Technologie |
|--------|------------|
| Frontend | Next.js (App Router) |
| Databáze | Neon (serverless Postgres) |
| Hosting | Railway |
| AI | Claude API (Anthropic) |
| Auth | Magic link (email) |
| Email | Brevo API |

Žádný Vercel tentokrát — Railway mi dává víc kontroly nad serverem a persistent storage. Neon jako databáze, protože serverless Postgres škáluje s load a branching je skvělý pro development.

## Filtrování a vyhledávání

Hlavní obrazovka portálu je tabulka s filtry. 17 665 záznamů musí být prohledávatelných za milisekundy.

**Filtry:**
- **Region** — kraj, okres, město
- **Velikost** — počet bytových jednotek (rozsah)
- **Kontakt** — má email / má telefon / má web
- **Zdraví** — bez exekuce / s exekucí
- **Správce** — vlastní výbor / externí správce
- **Signály** — nově založené, velké, problémové

Filtry jsou server-side (Neon query) s URL-based state — sdílitelné odkazy na konkrétní filtraci. Žádný client-side filtering na 17k záznamech.

## Signálové badges

Každé SVJ má barevné badges, co na první pohled řeknou kvalitu leadu:

- 🟢 **Kontakt ověřen** — email z 2+ zdrojů
- 🔵 **Velké SVJ** — 50+ bytů
- 🟡 **Nové SVJ** — založeno v posledních 2 letech
- 🔴 **Exekuce** — aktivní exekuční řízení
- ⚪ **Externí správce** — řídí správcovská firma

Prodejce vidí na první pohled: zelené + modré = kvalitní lead. Červené = pozor. Jednoduché, vizuální, efektivní.

## AI asistent

Chatbot přímo v portálu — Claude API s kontextem dat. Můžeš se zeptat:

- "Najdi SVJ v Praze 6 s víc než 100 byty a emailem"
- "Kolik SVJ v Brně má exekuci?"
- "Shrň mi tohle SVJ" (na detail stránce)
- "Jaké SVJ bys doporučil pro nabídku zateplení?"

Claude vidí strukturu dat a generuje SQL-like dotazy, které se přeloží na reálné filtry. Odpovídá přirozeně — ne jako databázový terminál.

Tohle je ta feature, kvůli které lidi zůstávají. Místo klikání na filtry prostě řekneš co chceš.

## Magic link autentizace

Žádná registrace, žádné heslo. Zadáš email → dostaneš odkaz → jsi přihlášený. Token v cookie, expirace 7 dní.

Proč magic link?
- **Nižší bariéra** — žádné vymýšlení hesla
- **Bezpečnější** — žádná hesla k ukradení
- **Jednodušší implementace** — žádné forgot password flow

Brevo API posílá emaily. Fallback na SMTP pokud Brevo nefunguje.

## Detail SVJ

Každý záznam má detail stránku s:

- **Základní info** — název, IČO, adresa, datum vzniku
- **Kontakty** — předseda, email, telefon, web (s confidence score)
- **Signály** — badges, exekuce, počet bytů
- **Mapa** — poloha z RÚIAN
- **Historie** — kdy byl záznam naposledy aktualizován
- **AI shrnutí** — Claude vygeneruje jednořádkové shrnutí z dat

## Co funguje

**Rychlost.** Neon Postgres s indexy na všechny filtrované sloupce. Filtrovaný dotaz na 17k záznamech: pod 50ms. Stránka se renderuje server-side, žádné loading spinnery na hlavním obsahu.

**UX.** Prodejce přijde, vyfiltruje si cílovou skupinu, exportne kontakty, osloví. Celý flow: 2 minuty. Dřív: hodiny googlování.

**AI asistent.** Lidé, co nejsou tech-savvy, prostě napíšou co chtějí. Claude to přeloží na filtry. Demokratizace přístupu k datům.

## Kde jsme teď

Portál je **funkční ale rozpracovaný**. Co zbývá:

- **Validace dat** — probíhá čištění a křížová kontrola. 17k záznamů, spousta edge cases.
- **Stripe integrace** — platební brána pro přístup k placeným datům. Free tier ukáže strukturu, platící zákazník vidí kontakty.
- **Rozšíření regionů** — teď Praha a Středočeský kraj. Celá ČR je na roadmapě.

## Co dál

**White-label.** Portál je připravený na multi-tenant — každý zákazník vidí jen svůj segment dat. Logo, barvy, doména — vše konfigurovatelné.

**Automatické aktualizace.** Pipeline teď běží manuálně (batch). Cíl: automatický re-scrape jednou týdně + detekce změn (nový předseda, nová exekuce).

## Technické rozhodnutí

**Proč Railway místo Vercel?** Potřebuji persistent server pro pipeline scripty, cron joby a delší běhy. Vercel je skvělý na Next.js frontend, ale pro backend pipeline s Scrapy je Railway lepší fit.

**Proč Neon místo Supabase?** Neon je čistý Postgres bez přidané magie. Branching pro development, serverless scaling, žádný vendor lock-in. Supabase by přidal auth a storage, ale magic link jsem implementoval sám za odpoledne.

**Proč Claude místo GPT pro chatbot?** Nižší halucinace. Když chatbot odpovídá na dotazy o reálných datech, nemůže si vymýšlet. Claude je konzistentněji přesný.
